# City data contract

City artifacts use schema version 2. A manifest identifies the city, timezone, bounds, attribution, backend and versioned artifact. The Vancouver Milestone 1 backend is `json-tuple-pack` at `data.pack`.

```json
{ "schemaVersion": 2, "sourceSnapshot": {}, "evidence": { "schemaVersion": 1, "sources": [], "records": [] }, "species": [], "trees": [], "treeMeasurements": {}, "foodSites": [], "trails": [] }
```

Each tree tuple has exactly nine fields: `[id, latitude, longitude, speciesIndex, heightM, diameterCm, canopySpreadM, address, sourceId]`. IDs are stable strings; coordinates are WGS84; height and canopy use metres; diameter uses centimetres. Missing measurements are `null`, never zero-filled. `canopySpreadM` is null in Milestone 1. Invalid coordinates or units and duplicate IDs are rejected before artifact generation.

Species metadata is embedded in the city artifact. The top-level `evidence` registry is schema v1 and contains deterministically ID-sorted `sources` and reviewed `records`; each source records canonical and licence URLs, version/retrieval/geographic/taxonomic scope, cultivar support, explicit reuse/redistribution/modification/derivative permissions, attribution, and supported claim kinds. Each record carries a stable taxon ID, its exact scientific name, optional exact cultivar, source record URL, source statement, caveats, review status/date, and supported structured claims. `edibility.status` is `unknown` unless a reviewed exact-taxon record supports it; species metadata never implies harvesting permission or safety. A reported-edible/not-edible claim requires stable evidence-record IDs; a bloom/harvest claim requires the same plus a valid month mask, while bloom also requires a semantic colour. Exact matching is normalized scientific-name plus exact cultivar (including both values being absent); common names, fuzzy names, genus fallback, and cultivar fallback do not enrich. Conflicting claim records resolve to unknown. `treeMeasurements` is keyed by tree ID and currently empty. When a canopy measurement exists, its `canopy` provenance is `measured` or `estimated`; an estimate also records its method. Missing canopy spread remains `null`, never zero-filled. `foodSites` contains distinct site-level records with stable site IDs, point location, reported food-tree count/varieties, jurisdiction/steward details, and `access.status: "unknown"`. It must never be converted to individual tree specimens. `trails` is an empty top-level collection until reviewed trail work begins. No standalone `species.json` is emitted.

`trails` contains only human-reviewed waypoint collections. `trailMembership` is the deterministic reverse map from a tree ID to compiled trail IDs and must exactly match `trails`. Candidate packets are non-public build reports, never trail records. A reviewed trail requires stable ID/name, city and snapshot provenance, reviewed waypoint order, existing-tree members, bounded export anchors whose coordinates match their member trees and city bounds, waypoint-oriented narrative, identified human reviewer/date/status, and explicit `unknown` accessibility and pedestrian-plausibility fields unless a later approved evidence contract supports something narrower. A waypoint sequence and Haversine span are not walking routes or walking distances.
