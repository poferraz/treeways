# v2.0.0 release gates

v2.0.0 is not release-ready unless every required CI job is green against the production build.

| Gate | Command | Evidence |
| --- | --- | --- |
| Static and data verification | `npm run check && npm run test:unit && npm run city:validate && npm run check:bundle` | CI log |
| Browser functionality | `npm run test:e2e` | Playwright HTML report, trace/video on failure |
| Browser compatibility | `npm run test:browser` | Chromium, Firefox, and WebKit report |
| Accessibility | `npm run test:a11y` | Axe report; no serious or critical violations |
| Offline behavior | `npm run test:offline` | Cached reload coverage and explicit routing limitation |
| Responsive visuals | `npm run test:visual` | Approved mobile, tablet, and desktop snapshots |
| Performance | `npm run test:performance` | Three-run Lighthouse CI report and scores |

## Performance thresholds

Measurements run against `dist/` with Lighthouse CI, three times per commit. The median must meet:

- Performance: 0.50 or higher (MapLibre is intentionally included in the production path).
- Accessibility, best practices, and SEO: 0.90 or higher.
- Service worker and offline audits: passing.

The bundle budget remains a separate hard gate. Any approved score or screenshot baseline change must be documented in its pull request.

## Manual release evidence

Before publishing, attach the successful workflow URL and visually inspect the approved screenshots at 320 px mobile, 768 px tablet, and 1280 px desktop. Also check 200% browser text zoom and a landscape phone manually; those are intentionally not represented by browser zoom in screenshot baselines. A first-ever offline visit cannot load a web application before its service worker has been installed; release content must state that one connected visit is required to enable offline use.
