# v2.1.0 Milestone 3-A candidate and trail-review gate

**Status:** M3-A complete; M3-B paused for explicit human review. Review tool ready at [`docs/m3-b/review-tool.html`](../../m3-b/review-tool.html). Last code-touch: 2026-07-15.

## Premises

- **[P1] Repository state.** `master` remains two commits ahead of `origin/master`; the existing dirty Milestone 1/2 working set was preserved. No staging, commit, publish, deployment, or external account action occurred.
- **[P2] Pipeline.** A checksum-verified snapshot flows through `normalizeCity()`, then `buildCityPack()`, generated/report validation, benchmark validation, and deterministic verification. The selected JSON tuple backend, exact filtering semantics, rejection accounting, and 5,600,000-byte gzip cap remain unchanged.
- **[P3] Trail contract.** M3 requires generated candidates to remain distinct from reviewed waypoint trails. Only an identified human reviewer can approve a final name, ordered membership, export anchors, waypoint-oriented narrative, and unknown-preserving access/pedestrian notes.
- **[P4] Supported candidate inputs.** Tree ID, WGS84 coordinates, valid reported height/diameter/canopy fields, and the shared `classifyGiant()` result. The current Vancouver inventory has no canopy values, so giant candidates are height-derived.
- **[P5] Unsupported inputs.** Forage, bloom, harvest, food access, harvesting permission, safety, accessibility, pedestrian plausibility, and walking route/distance claims. These are never candidate inputs or output claims.
- **[P6] Baseline.** Before M3-A, the public artifact had 185,307 accepted trees, 538 rejects, zero trails, and SHA-256 `df58bd2f306791851be3e1ed76a12cfe14791ae1cb92220ccd9fe23c9bdbbfd9`.

## Execution traces

- **[T1] Candidate entry and ID.** `buildCityPack()` calls `generateCandidateReviewPacket()` after writing the artifact. `scripts/city/trails.js` filters only through `classifyGiant()`, groups fixed 0.02-degree cells, derives an ID from SHA-256 of city, source snapshot hash, theme, cell, and ordered waypoint IDs, and sorts final candidates by ID. No timestamps or randomness enter output.
- **[T2] Order, distance, and score.** Each group ranks solely by `heightM + diameterCm/100 + canopySpreadM/10`, then produces a nearest-neighbour waypoint sequence with ID tie-breaks. It reports Haversine straight-line waypoint distance in metres and explicitly marks it as not walking distance or pedestrian routing.
- **[T3] Review packet separation.** The packet is written only to `data/cities/vancouver/reports/trail-candidates.json`, includes `NOT HUMAN REVIEWED`, unsupported fields, unresolved unknowns, and required human decisions. It is not read by runtime code and cannot populate `trails`.
- **[T4] Reviewed-source compiler.** `normalizeCity()` reads `trails-review.json` and calls `compileReviewedTrails()`. Non-empty records require city/snapshot/generator provenance, unique IDs, an identified non-AI human reviewer, review date/status, ordered existing tree IDs, bounded export anchors matching exact tree coordinates and city bounds, a waypoint-oriented narrative, and `unknown` accessibility/pedestrian fields.
- **[T5] Compilation and reverse membership.** The compiler copies only validated reviewed fields, sorts trails and membership IDs deterministically, and emits `trailMembership` from compiled `trails`. `validateTreePack()` and `validateCity()` reject membership that is not an exact reverse mapping.
- **[T6] Artifact and determinism.** `verify-city-determinism.js` rebuilds twice and compares both artifact and candidate-packet bytes. `city:validate` verifies normalized/artifact drift, count/rejection reports, food-site access unknowns, membership, and benchmark identity.
- **[T7] Tests.** `tests/city-pipeline.test.js` proves byte-stable candidates, giant classifier use, no season/forage candidates, straight-line wording, deterministic score components, human-review enforcement, duplicate ID/missing-tree/out-of-bounds-anchor/invalid-order rejection, raw candidate rejection, empty reviewed input, deterministic compilation, and exact reverse membership. Existing city/domain tests retain Milestone 1/2 safety checks.

## Candidate review packet

The generated packet is [`trail-candidates.json`](../../data/cities/vancouver/reports/trail-candidates.json). It contains neutral generated IDs and labels only; it is not a published trail set. Human review must decide whether any candidate should become a trail and must provide the required reviewed fields in [`trails-review.json`](../../data/cities/vancouver/trails-review.json).

## M3-B stop condition

M3-B requires explicit human approval of specific candidate IDs plus a reviewed source record for each approved trail. Until then, `trails-review.json` remains empty, the public artifact contains `trails: []` and `trailMembership: {}`, and no candidate is curated, reviewed, accessible, safe, walkable, or routable.
